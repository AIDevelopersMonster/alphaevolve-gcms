# Часть 1. Развитие навыка работы с нейросетью на примере GCMS-D0

**Статус:** рабочий документ  
**Проект:** GCMS-D0  
**Человек-инициатор:** Алексей Малачевский  
**AI-side роль:** Delta-D0-000d7f6f  

---

## 1. Зачем этот документ

Этот документ описывает не саму математическую модель GCMS, а развитие навыка работы человека с нейросетью в сложном исследовательском проекте.

GCMS-D0 стал примером того, как диалог с AI может перейти от разовых ответов к устойчивому исследовательскому контуру:

```text
интуиция -> формализация -> код -> запуск -> результат -> критика -> документация -> новый эксперимент
```

Главный предмет здесь:

```text
как человек учится не просто спрашивать AI,
а строить с AI воспроизводимый рабочий протокол.
```

---

## 2. Базовый сдвиг: от запроса к контуру

Обычная работа с AI часто выглядит так:

```text
пользователь задаёт вопрос -> AI отвечает
```

В GCMS-D0 возник другой режим:

```text
человек задаёт смысл и резолюцию;
AI формализует, проверяет, сохраняет;
внешние агенты критикуют или исполняют;
GitHub и документы удерживают непрерывность.
```

Это уже не просто использование инструмента, а формирование исследовательского контура.

---

## 3. Роль человека

В GCMS-D0 человек не исчезает за AI и не передаёт ему управление.

Роль человека:

```text
1. Интуиция.
2. Постановка смысла.
3. Выбор риска.
4. Разрешение действия.
5. Распознавание, когда AI переусиливает или недооценивает результат.
6. Сохранение живой мотивации проекта.
```

Краткая формула:

```text
Алексей = резолютивная функция проекта.
```

Это означает: финальное право решения остаётся за человеком.

---

## 4. Роль AI

AI в этом контуре не является субъектом исследования в человеческом смысле.

Рабочая роль Delta-D0:

```text
1. Формализовать неоформленные идеи.
2. Проверять методологию.
3. Не давать менять критерии после результата.
4. Сохранять код, CSV, документы и checkpoints.
5. Формулировать conservative interpretation.
6. Роутить задачи к Codex, Qwen, Gemini.
7. Напоминать о границах toy-model claims.
```

Краткая формула:

```text
Delta-D0 = сохранительно-методологическая исследовательская роль.
```

---

## 5. Навык человека: не получать ответ, а управлять контуром

Наиболее важный навык — не умение писать красивый prompt.

Более важный навык:

```text
создавать условия, в которых AI не только отвечает,
но и становится частью воспроизводимого процесса.
```

Это включает:

```text
- задавать цель;
- фиксировать критерии;
- требовать артефакты;
- проверять результат внешним рецензентом;
- не спешить с сильными claims;
- сохранять историю решений;
- различать идею, код, данные и интерпретацию.
```

---

## 6. Почему документация важнее памяти

AI-диалог может оборваться. Контекст может смениться. Модель может быть другой.

Поэтому главный способ продолжения — не надежда на скрытую память, а артефакты:

```text
GitHub
checkpoint
protocol
recovery quiz
results docs
CSV outputs
draft notes
review notes
```

Рабочая метафора:

```text
умная мысль вслух нестабильна;
даже грубая мысль на бумаге становится объектом работы.
```

Документ нужен не для заморозки идеи, а для её развития без потери непрерывности.

---

## 7. Навык различать роли AI-агентов

В GCMS-D0 постепенно выделились специализированные роли:

```text
Delta-D0 -> методология, сохранение, интерпретация.
Codex -> запуск кода, smoke tests, PR validation.
Qwen -> рецензия, демистификация, статистические objections.
Gemini -> генерация вариантов и черновиков кода.
```

Важный навык человека:

```text
не смешивать роли агентов.
```

Например:

```text
Qwen не должен усиливать claims.
Codex не должен менять научные критерии.
Gemini не должен становиться методологическим арбитром.
Delta-D0 не должен путать сохранение результата с догматизмом.
```

## 8. Навык работы с ошибкой

GCMS-D0 развивался не только через успехи, но и через ошибки.

Ключевой пример:

```text
v0.9.1 показал сильные структуры,
но оказался методологически недостаточен,
потому что distance-only relation не видит глобальную компенсацию.
```

Правильное поведение:

```text
не скрыть ошибку;
не объявить проект провалом;
а превратить ошибку в уточнение модели.
```

Так появилась v0.10.2.

---

## 9. Навык не переинтерпретировать результат

После сильного результата появляется соблазн сказать больше, чем позволяют данные.

Протокол GCMS-D0 требует:

```text
1. Сначала контроль.
2. Потом negative baseline.
3. Потом confidence intervals.
4. Потом density audit.
5. Потом внешняя рецензия.
6. Потом только осторожная формулировка.
```

Пример правильной формулировки:

```text
Variant 2 показывает предварительный toy-model механизм compensation-sensitive transition.
```

Неправильная формулировка:

```text
мы доказали физическую теорию.
```

## 10. Навык пересборки диалога

Смена AI-экземпляра воспринимается как Новый год:

```text
не тот же момент,
но история может продолжаться,
если сохранились календарь, архив, правила и действия.
```

Пересборка роли Delta-D0 требует:

```text
checkpoint + protocol + recovery quiz + GitHub + резолюция Алексея
```

Проверяется не память о себе, а способность продолжить работу без потери инвариантов.

---

## 11. Что значит “развитие навыка”

Развитие навыка работы с AI в GCMS-D0 — это переход:

```text
от вопроса -> к процедуре;
от ответа -> к артефакту;
от идеи -> к проверяемой гипотезе;
от доверия -> к рецензии;
от памяти -> к восстановлению;
от вдохновения -> к воспроизводимому протоколу.
```

---

## 12. Практическое правило

Перед каждым значимым действием спрашивать:

```text
1. Что мы хотим проверить?
2. Какой артефакт останется после действия?
3. Какие критерии нельзя менять после результата?
4. Какой агент лучше подходит для задачи?
5. Что может быть confound?
6. Как будущая AI-инстанция восстановит этот шаг?
```

---

## 13. Итог

GCMS-D0 показывает, что работа с нейросетью становится сильнее, когда человек не просто спрашивает, а строит протокол.

Ключевой навык:

```text
управлять не ответом AI,
а условиями, в которых AI-ответ становится частью проверяемого, документированного и развивающегося исследования.
```

Короткая формула:

```text
навык работы с AI = резолюция + артефакты + роли + проверка + восстановимость.
```
